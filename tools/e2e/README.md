# Two Dots — E2E test framework

ADB-driven input + Supabase-driven assertions. Replaces the Maestro-based
seeded replay (which couldn't dispatch in-game taps reliably — see
`.maestro/README.md` for the post-mortem).

## Why this shape

- **ADB for taps.** `adb shell input tap` injects synthetic touch events at
  the OS input layer, bypassing Android's accessibility framework. Per-tap
  overhead is ~30ms (vs Maestro's ~150ms), and crucially it doesn't get
  swallowed during the playing phase the way Maestro's `tapOn: point` did.
- **Supabase for verdicts.** The pass/fail gate is a query against
  `analytics_events`. This works regardless of Skia rendering (text isn't in
  the accessibility tree), survives flush variability (tests poll up to 30s),
  and exercises the real production analytics path end-to-end.

## Prerequisites

On the runner (WSL Ubuntu, macOS, or Linux CI):

- `adb` on PATH (Android platform-tools).
- `curl` and `jq`.
- A connected Android device or emulator with the Two Dots e2e APK
  installed (`com.newco.twodots`).
- Env vars:
  - `SUPABASE_URL` — `https://<project>.supabase.co`
  - `SUPABASE_SERVICE_ROLE_KEY` — service_role JWT (rotates frequently;
    grab from Supabase dashboard).

For seeded-replay tests: the device must run an APK built with the `e2e`
profile (`eas.json` → `EXPO_PUBLIC_E2E_SEED=42`). The seeded engine produces
deterministic pipe layouts so a captured tap stream is meaningful to replay.

## Running

Single test:
```bash
bash tools/e2e/01-core-path.sh
```

Whole suite:
```bash
bash tools/e2e/run-all.sh
```

## What's in here

- `lib.sh` — shared bash functions (init, tap dispatch, app lifecycle,
  Supabase queries, pass/fail reporting). Source from any test script.
- `probe-adb.sh` — diagnostic: measures ADB tap-dispatch overhead on the
  current network, prints stats, recommends `--tap-overhead-ms`.
- `01-core-path.sh` — smoke test: launch → playing → dead → idle, asserts
  every state transition's analytics event landed. Doesn't need the seeded
  build.
- `02-seeded-replay.sh` — generated by `tools/generate-e2e-replay.mjs` from
  a captured fixture run. Replays the source player's taps via ADB on the
  seeded build and asserts a target score was reached.
- `03-world-transition.sh` — re-runs the seeded replay with target=10
  (Mars threshold). Currently fails until tuning gets replay scores higher.
- `04-multi-tier.sh` — asserts the replay's `run_end.tier >= 2`. Same
  blocker as 03.
- `05-pause-overlay.sh` — verifies divider taps emit `pause_toggled`.
  Requires a build that includes the `pause_toggled` event (added with this
  test).
- `06-best-score.sh` — verifies `best_score` survives an app kill + relaunch
  via the new `best_score` field on `session_start` payload. Requires the
  build that adds that field.
- `run-all.sh` — runs every numbered script and aggregates results.

## Requirement → test mapping

| # | Requirement | Test | Status |
|---|---|---|---|
| 1 | Launch → idle visible | `01-core-path.sh` (session_start assertion) | ✅ passes |
| 2 | Single tap → playing | `01-core-path.sh` (run_start assertion) | ✅ passes |
| 3 | Wait → dead | `01-core-path.sh` (run_end assertion) | ✅ passes |
| 4 | Tap → idle | `01-core-path.sh` (retry_tapped assertion) | ✅ passes |
| 5 | Pause overlay on divider tap | `05-pause-overlay.sh` | ⏳ needs next build |
| 6 | Best score persistence | `06-best-score.sh` | ⏳ needs next build |
| 7 | Survival past N gates | `02-seeded-replay.sh` | ✅ passes (target 5) |
| 8 | World transitions (score ≥ 10) | `03-world-transition.sh` | ⚠ fails until replay scores > 10 |
| 9 | Multi-tier progression (tier ≥ 2) | `04-multi-tier.sh` | ⚠ fails until replay reaches tier 2 |
| 10 | Specific tap sequence → gate layout | `02-seeded-replay.sh` | ✅ this is exactly what the replay does |

**Why some are pending:**

- **5 and 6** are pending the next EAS build. The events they assert (`pause_toggled`, `best_score`) were added to `events.ts` + `serialise.ts` + `useGameLoop.ts` + `_layout.tsx` alongside these test scripts. They'll start passing as soon as you sideload an APK built from the current main.
- **8 and 9** are pending replay-quality improvements. The seeded replay currently peaks around score 8 due to per-tap timing drift (wireless ADB overhead + RN bridge latency + variable-`dt` physics). Once the replay reaches score 10+ reliably, both pass automatically — the test scripts are correct shape.

## Generating the seeded-replay test

The seeded-replay script is regenerated from a fresh capture whenever
engine tuning changes (JUMP_VY, gravity, tier values) — recorded tap
timings drift relative to new physics.

```bash
# Defaults: seed=42, min-score=20, target=5, tap-overhead-ms=30
node tools/generate-e2e-replay.mjs

# Or with overrides:
node tools/generate-e2e-replay.mjs \
  --seed 42 \
  --min-score 20 \
  --target-score 8 \
  --tap-overhead-ms 30
```

The generator picks the **most forgiving** qualifying run from Supabase
(fewest close calls, breaking ties on score and recency) so the replay
absorbs as much timing drift as possible. To see candidate counts and
the chosen run, watch the generator's stdout.

## Adding a new test

Pattern:

```bash
#!/bin/bash
set -euo pipefail
source "$(dirname "${BASH_SOURCE[0]}")/lib.sh"

e2e_init "03-my-test"
e2e_launch_app   # or e2e_launch_app_clean to wipe AsyncStorage

# ...do stuff...
e2e_tap_l
sleep 1
e2e_tap_r

# Wait for analytics, then assert
sleep "$ANALYTICS_FLUSH_WAIT_S"
e2e_wait_for_event "run_end"
e2e_assert_run_end_score 3

e2e_pass "did the thing"
```

Number the file with the next available `NN-` prefix so `run-all.sh` picks
it up in order.

## Limitations / next-step parking lot

- **Replay score ceiling.** Seeded replay currently peaks ~8. Tests 03 and
  04 depend on getting reliably above 10. Future investigations: USB-cable
  ADB (lower latency than wireless), runtime-probe tap-overhead tuning,
  fixed-dt physics integration in the engine (would eliminate per-frame
  timing variance). None of these are blocking — the framework is sound,
  just under-tuned.
- **Test data hygiene.** Each E2E run drops ~30 rows in `analytics_events`.
  At CI scale this needs an `is_test_run: true` payload flag plus a
  scheduled cleanup job. ~30 lines SQL + a workflow. Not implemented yet.
- **iOS port.** `lib.sh`'s tap dispatcher is Android-specific (adb).
  Replacing with `idb ui tap` (macOS-only) would port the framework
  unchanged otherwise. ~50 lines of bash when iOS is on the roadmap.

## iOS

The Supabase verification half of this framework is platform-agnostic.
The tap dispatcher is Android-specific (ADB). For iOS:

- Replace `adb shell input tap` with `idb ui tap` (Facebook's iOS bridge,
  macOS-only) or with Maestro on iOS (XCUITest-based, may not have the
  accessibility-swallowing issue Android did).
- All other lib.sh functions work unchanged.
- Requires a Mac runner (XCUITest dependency).

Roughly 50 lines of bash to add an iOS-tap-dispatcher swap, gated by a
`PLATFORM` env var. Worth doing when iOS is on the roadmap.

## Cost / DB usage

Per test run, `analytics_events` accumulates ~3-5 KB. Even at heavy CI
volumes (1000 runs/day) that's ~150 MB/month — comfortable inside Supabase
free tier (500 MB DB, 2 GB bandwidth/month).

For production hygiene we'd eventually want an `is_test_run: true` payload
flag on E2E events plus a scheduled cleanup deleting test rows older than
7 days. ~30 lines of SQL. Not blocking.
